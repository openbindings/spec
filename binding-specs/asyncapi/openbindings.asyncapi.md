# `openbindings.asyncapi` Binding Specification

## 1. Status and identifier

This document is the normative text for the binding specification identifier **`openbindings.asyncapi@1`**, published by the openbindings project as its defining authority. The identifier is exact and opaque per core [OBI-B-01](../../openbindings.md#104-binding-specification-rules): no range or normalization semantics attach to it. An incompatible change to this specification publishes `openbindings.asyncapi@2` ([OBI-B-03](../../openbindings.md#104-binding-specification-rules)); compatible clarification may revise this document in place. It publishes with the OpenBindings core 0.2.0 change set, and reference tooling adopts the identifier — replacing the pre-bindingSpec `asyncapi@…` tokens — in the same coordinated change.

The key words "MUST", "MUST NOT", "REQUIRED", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [BCP 14](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174) when, and only when, they appear in all capitals.

## 2. Scope and incorporated authorities

This is the **openbindings project's** binding specification for [AsyncAPI](https://www.asyncapi.com/docs/reference/specification/v3.0.0) documents. It is published under this project's authority, not by or for the AsyncAPI Initiative: AsyncAPI 3.0.0 and the AsyncAPI HTTP binding 0.3.0 and WebSockets binding 0.1.0 are incorporated by reference and remain authoritative over the artifact — document structure and reference resolution, channels, addresses and parameters, operations and their actions, messages and content types, servers and their variables, protocol `bindings` objects, and security scheme declarations. Plain HTTP semantics are [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110)'s; WebSocket mechanics are [RFC 6455](https://www.rfc-editor.org/rfc/rfc6455)'s. This specification defines only the OpenBindings overlay: how a source carries or addresses an AsyncAPI document, how a binding selects an operation, and how caller-facing operation values correspond to messages on the wire. For stability of this exact identifier, an HTTP or WebSockets binding object that omits its upstream `bindingVersion` is interpreted at the incorporated revision named here — 0.3.0 or 0.1.0 respectively — rather than against whatever a mutable upstream page may later call “latest”; an explicit different binding version is outside revision 1 and is refused.

**Normative upstream snapshot.** AsyncAPI 3.0.0 is [`spec/asyncapi.md`](https://github.com/asyncapi/spec/blob/d78dcea70c9b094a3df72f9a4e811828cec778cc/spec/asyncapi.md) at official tag commit [`d78dcea70c9b094a3df72f9a4e811828cec778cc`](https://github.com/asyncapi/spec/tree/d78dcea70c9b094a3df72f9a4e811828cec778cc). The HTTP 0.3.0 and WebSockets 0.1.0 authorities are [`http/README.md`](https://github.com/asyncapi/bindings/blob/aebfb0207d9faba7b3d05ffe8f2f2a7cfd0b5431/http/README.md) and [`websockets/README.md`](https://github.com/asyncapi/bindings/blob/aebfb0207d9faba7b3d05ffe8f2f2a7cfd0b5431/websockets/README.md) at official bindings commit [`aebfb0207d9faba7b3d05ffe8f2f2a7cfd0b5431`](https://github.com/asyncapi/bindings/tree/aebfb0207d9faba7b3d05ffe8f2f2a7cfd0b5431). String carriage uses [YAML 1.2.2](https://yaml.org/spec/1.2.2/). Rendered documentation and later repository revisions are informative unless this binding specification deliberately advances a pin.

Revision 1 binds **`receive` operations over `http`/`https`** and **`receive` or `send` operations over `ws`/`wss`**. A standalone `send` operation over `http`/`https` is excluded: AsyncAPI's HTTP operation binding declares request mechanics, but it does not declare a counterparty subscription, listening lifecycle, or SSE framing that an OpenBindings invoker can derive. Revision 1 refuses that cell instead of inventing one. Other AsyncAPI protocols — kafka, mqtt, amqp, and the rest — are likewise **excluded**: each exclusion is a definition, not an open item, and a future revision or a more specific binding specification may bind it. Resource policy — delivery-unit size caps, subscription buffering bounds, idle timeouts — is consumer and implementation policy, outside this document (with one integrity floor at [§9.4](#94-classification)).

## 3. Accepted source representations

This specification accepts **AsyncAPI 3.0.0 documents exactly**, in two carried representations (a later edition is adopted by compatible revision of this document, never sight-unseen):

- **Object** `content`: the parsed AsyncAPI document as a JSON object.
- **String** `content`: the document's source text. This specification pins the string grammar — its own pin: string content parses as **YAML 1.2.2** under the AsyncAPI specification's JSON-compatibility recommendations — this specification's pin, not an AsyncAPI mandate — of which JSON is a valid subset, so one grammar covers both spellings deterministically. Duplicate mapping keys are refused loudly.

Discrimination is the artifact's own **`asyncapi` field**: the exact value `3.0.0` is accepted; every other value — including later 3.0 patches and AsyncAPI 2.x — is refused loudly at load (**ASYNC-P-01**). No other representation is defined.

## 4. `location`

A source's `location`, when present, is an **absolute URI addressing the AsyncAPI document itself** (**ASYNC-D-02**) — `https://example.com/asyncapi.json`, `file:///srv/api/asyncapi.yaml`. Dereferencing it yields an accepted representation ([§3](#3-accepted-source-representations)). A bare filesystem path is a relative reference in form and is not a conformant `location` (core [OBI-D-05](../../openbindings.md#102-document-rules)). This family is artifact-located: connection targets come from the artifact's `servers`, not from `location`.

## 5. `content`

A source's `content`, when present, MUST be one of the two accepted representations of [§3](#3-accepted-source-representations) (**ASYNC-D-01**). No other JSON type is an accepted `content` value under this specification.

## 6. Composition

When `content` is present it is the artifact the processor interprets, per the core's content-primacy floor ([§5.4](../../openbindings.md#54-sources)).

A co-present `location` is the document's origin, and it serves as the embedded artifact's **base URI**: references internal to the embedded document (relative `$ref`s) resolve against it per the AsyncAPI specification's own reference-resolution rules, exactly as they would had the document been retrieved from that address. The base travels in the OBI, so resolution is identical everywhere; the OBI's own retrieval URI is never a base (core [§7](../../openbindings.md#7-reference-resolution)). Embedded content with **no** co-present `location` has no base and MUST be self-contained (bundle before embedding). Self-containment is RECOMMENDED even with a `location`. For a `location`-only source, the document's base URI is the `location`.

## 7. `ref`

A binding's `ref` is REQUIRED and MUST be a JSON Pointer of the form `#/operations/<operation-key>`, addressing an entry of the artifact's `operations` map (**ASYNC-D-03**) — one spelling; a bare operation key without the pointer prefix is not conformant, and keys containing `/` or `~` carry [RFC 6901](https://www.rfc-editor.org/rfc/rfc6901) escaping. That one spelling is **verbatim**: the `#/` prefix is pointer notation, not URI processing — no percent-decoding is ever applied, and a `ref` matches its target byte-for-byte, RFC 6901 escapes included, per the core's literal-form doctrine (core [§7](../../openbindings.md#7-reference-resolution)). A percent-encoded rendering (`#/operations/tasks%2Fcreate` for the key `tasks/create`, whose one spelling is `#/operations/tasks~1create`) is not that spelling and denotes nothing. An entry that is a Reference Object resolves through it before the operation-object test, per the artifact's own reference rules. Only operations are addressable: channels, messages, servers, and the artifact as a whole are not binding targets, and this family defines no whole-artifact invocation. A well-formed `ref` whose operation is absent from the artifact makes the binding unresolvable; verification requires the artifact, per the core's partial-verification posture.

## 8. Target and interaction

**Perspective — incorporated, then bounded.** An AsyncAPI artifact describes the application it documents: `send` declares that the described application sends to the channel, and `receive` that it expects to receive from it — the AsyncAPI specification's own perspective rule. An invocation acts as the **counterparty only where the declared protocol supplies an interaction the invoker can initiate faithfully**: it publishes what a `receive` operation expects, and over WebSocket it subscribes to what a `send` operation emits. This specification never reverses the artifact's action, assumes a missing opposite application, or manufactures a transport interaction merely from the word `send` (**ASYNC-P-02**).

The binding target is the addressed operation's channel, reached through a resolved server and expanded address ([§9.2](#92-configuration-points)). The interaction shape is the operation's `action` crossed with the **selected artifact server's declared `protocol`** — statically decidable before dispatch. A complete connection-URL replacement changes only where that selected server is reached; its scheme MUST equal the selected server's declared protocol. An override never changes an HTTP operation into a WebSocket operation or vice versa.

| `action` × protocol    | Interaction                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `receive` × http/https | **Unary publish**: exactly one input value, dispatched as the selected operation message with the request method the artifact's http operation binding declares. An absent method makes the binding unresolvable — revision 1 does not infer POST. When the operation declares a `reply`, a non-empty response body decodes to one output under the reply message selected by the final status ([§9.3](#93-decode)); without a declared reply, and for every empty response body (202/204 acknowledgments included), the invocation emits no output value — an undeclared response body is not promoted into an OpenBindings result |
| `receive` × ws/wss     | **Client-streaming publish**, only when the operation declares no `reply`: one input value per complete WebSocket data message, the caller closes input; zero output values. Fragmentation is reassembled by RFC 6455 before the operation boundary. Complete data messages the server sends are outside the selected operation and are ignored. A `receive` operation that declares `reply` is refused at binding resolution in revision 1; discarding its declared replies would change the artifact, while binding its reply channel requires a request/reply session model this revision does not define                        |
| `send` × http/https    | **Excluded**: the binding is refused before dispatch. An HTTP method or `send` action does not by itself declare a counterparty subscription, server lifecycle, or SSE framing                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `send` × ws/wss        | **Server-streaming subscription**: no input values; each complete received WebSocket data message emits one output value after RFC 6455 fragmentation reassembly. The subscription ends when the transport closes or the consumer cancels. Sending data messages would require a separately selected `receive` operation and is not inferred from this binding                                                                                                                                                                                                                                                                      |

Declared protocol **`bindings` objects are authoritative where they speak**, incorporated: an http operation binding's `method` selects the request method above and is required for the supported HTTP cell; its `query` schema governs HTTP query values. A websockets channel binding's `method`, `query`, and `headers` govern the upgrade request. Concrete query and header values are supplied through the required-when-needed **`protocolFields`** configuration point ([§9.2](#92-configuration-points)); they are never taken from the message payload or invented from JSON Schema annotations. Where an upstream protocol itself supplies a default, that default remains authoritative; this specification does not fill a missing HTTP method. One invocation is one HTTP publish, one WebSocket publish session, or one WebSocket subscription.

The revision 1 operation boundary carries a message's **payload**, not its AsyncAPI Message Object `headers`. If a selected input message or any possible subscription-output message declares `headers`, the binding is refused before transport establishment. If the reply message selected by an HTTP response declares them, that response is a protocol failure before output. Those declarations are part of the artifact and MUST NOT be silently discarded or approximated as payload fields. A future revision may define an explicit payload-and-headers envelope.

## 9. Operation-boundary correspondence

### 9.1. Input mapping and encoding

There is no field routing in this family: the input value is the **message payload, wholesale** (**ASYNC-P-03**). A publish operation must govern that payload with one operation message. When its resolved `messages` list has one member, that member selects itself; with several, the **`message`** configuration point is required to select one artifact-declared member; with no members, the binding is refused. Selection is by member of the artifact's list, not by validating or sniffing the caller value. A configuration choice outside that list is refused.

Input **encoding** follows the selected message's effective content type — the message's `contentType`, else the document's `defaultContentType` — and the **`encode`** point where the artifact leaves it absent:

- a **JSON-family** type (`application/json`, any `+json` suffix) serializes the value as JSON;
- **any other** declared type carries a **string** input value as its bytes, encoded per the type's `charset` parameter, defaulting to UTF-8 — a non-string value against a non-JSON type is refused before dispatch (a value that cannot ride the declared wire, not a routing choice made from the payload);
- with **no effective content type**, `encode` is required. Its choice is either the JSON lane above or the text lane above; invocation without that choice refuses before dispatch. This specification does not infer JSON merely because the OpenBindings value domain is JSON.

This is the boundary correspondence the value domain supports (core [§5](../../openbindings.md#5-document-model)): a JSON wire can carry every operation value and a character wire can carry a string value. Arbitrary **bytes** have no carriage here — an Avro, Protobuf, or other binary payload is unsendable because revision 1 defines no bytes boundary encoding. It is refused rather than assigned an OpenBindings-specific Base64 convention the artifact did not declare.

For HTTP, those encoded octets are the request body. For a WebSocket publish, AsyncAPI's incorporated WebSockets binding declares the handshake but no data-message opcode, while RFC 6455 distinguishes text and binary messages. The required **`websocketMessageType`** point therefore preserves that invocation-time choice instead of inventing a default: `text` sends one complete text message and is admissible only when the encoded payload is valid UTF-8; `binary` sends the same encoded octets as one complete binary message. Fragmentation is transport implementation detail and never changes the one-value/one-message boundary.

Presence per cell: a **publish** invocation (`receive`-action) requires an input value — absent input is refused before dispatch, because the input _is_ the message and this family defines no empty message. A WebSocket **subscription** takes no input; its input side is closed at establishment. Standalone HTTP `send` operations and operations with declared message headers are refused during binding resolution, before this presence rule is reached.

### 9.2. Configuration points

This family has **seven** named configuration points, consulted per-invocation configuration → consumer-level configuration → the artifact answer or binding rule below; a declined override falls through (**ASYNC-P-04**). Classification is not configurable ([§9.4](#94-classification)). A required point reached without a choice refuses before dispatch or establishment. No default depends on payload bytes:

| Point                    | Artifact answer or fallback                                                                                                                                                                                | Meaning                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **server**               | the sole bindable member of the operation's **effective server set**; no default when several remain                                                                                                       | Which declared server the connection targets. The effective set is the channel's declared `servers` subset when present and non-empty, else the document's `servers` map, filtered to protocols revision 1 binds. With several candidates, consumer configuration MUST select one or invocation refuses before dispatch. Configuration may supply its declared variables and may replace that selected member's connection URL only with one using the same scheme as its declared protocol; the selected member's artifact declarations and security still govern ([§9.5](#95-credentials)). No resolvable server is likewise a refusal. |
| **address**              | the channel's declared `address`; a parameter declaring `location` derives its value from the outgoing message where one exists, otherwise consumer-supplied parameter value, otherwise declared `default` | The channel's concrete address. Artifact-declared runtime expressions are authoritative and cannot be overridden. An absent or `null` address with no consumer-supplied whole address, an unsupported pre-connection runtime expression, or any expression left unresolved is a **pre-dispatch refusal**, never a guess (this specification does not assume the channel key is an address, and never dials literal braces).                                                                                                                                                                                                               |
| **message**              | the sole member of a publish operation's resolved `messages` list; required when several remain                                                                                                            | Which artifact-declared message governs one publish invocation. The point selects a list member, never a media type inferred from the input value. It is not consulted for subscription output or reply recognition.                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **protocolFields**       | omit every optional declared field; required declared fields have no fallback                                                                                                                              | Supplies string values for an HTTP operation binding's query schema or a WebSockets channel binding's query and header schemas. Values ride the handshake/request channels those bindings declare, never the message payload.                                                                                                                                                                                                                                                                                                                                                                                                             |
| **encode**               | the selected input message's effective content type; required lane choice when it has none                                                                                                                 | How the input payload becomes bytes, per [§9.1](#91-input-mapping-and-encoding). A choice cannot override a declared content type; it completes only an absent declaration.                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **websocketMessageType** | none; REQUIRED for a WebSocket publish                                                                                                                                                                     | Selects the RFC 6455 data-message opcode, `text` or `binary`, which the AsyncAPI WebSockets binding does not declare. It never changes content type or encoding. Subscription input is closed, and received messages carry their own opcode, so the point is not consulted there.                                                                                                                                                                                                                                                                                                                                                         |
| **decode**               | the single governing output message content type; required lane choice only when that declaration is absent                                                                                                | How each complete HTTP body or WebSocket data message becomes an output value — the decode **lane** (JSON vs text), per [§9.3](#93-decode). It fills an absent content-type declaration; it does not select among contradictory declarations. Envelope unwrapping (`{data}`/`{error}` conventions) is the operation's output transform's job ([§9.6](#96-transform-positions)), not a decode override.                                                                                                                                                                                                                                    |

**What each point accepts** (the semantics; the concrete carriage — the value a consumer hands to a point — is implementation surface, not pinned here, see below):

- **`server`** selects a member of the effective server set (implicitly only when the set has one), optionally supplying values for that server's own declared variables and optionally replacing its resolved target with a complete connection URL. A replacement changes the target, not the selected artifact server or its protocol: the replacement URL's scheme MUST equal the selected server's `protocol`, or the invocation refuses. A supplied variable name the selected server does not declare is refused, never ignored — a typo does not silently dispatch elsewhere. Supplying a member's variables is load-bearing rather than cosmetic: AsyncAPI declares a Server Variable's `default` OPTIONAL, so an undefaulted variable is satisfiable only by a supplied value (the [openapi sibling](../openapi/openbindings.openapi.md#93-servers-and-the-target-url) accepts the same supply at its server point).
- **`address`** supplies values only for channel parameters whose Parameter Object has no `location`, over declared defaults, or supplies the channel's whole address outright (the case AsyncAPI defines by declaring an `address` of `null` “unknown … generated dynamically at runtime”). A supplied parameter name the channel does not declare, or a supplied value for a parameter whose artifact declares `location`, is refused rather than ignored. A parameter with `location` follows AsyncAPI's runtime-expression semantics instead: for a `receive`-action publish, the expression is evaluated against the selected outgoing message, with `$message.payload` available and Message Object headers unavailable because revision 1 excludes them; a header-dependent or otherwise unavailable expression refuses before dispatch. A `send`-action subscription has no message before it must establish the target, so any address parameter with `location` makes that binding unresolvable in revision 1. Configuration never overrides a `location` result. Each substituted result, supplied value, or default MUST be a string admitted by the parameter schema; other value types are refused rather than assigned a new URI serialization.
- **`message`** selects one resolved member of the operation's own non-empty `messages` list. A sole member needs no configuration; several members require a choice. A channel message excluded from that operation's list is not an admissible choice.
- **`protocolFields`** supplies three disjoint maps as applicable: HTTP query, WebSocket upgrade query, and WebSocket upgrade headers. Each supplied name MUST be a property declared by the corresponding binding schema and each value MUST be a string admitted by that property schema. A required property with no supplied value refuses; optional unsupplied properties are omitted. JSON Schema `default` is an annotation and is not substituted. Revision 1 refuses a binding property whose schema cannot admit a string, because the AsyncAPI protocol binding defines no array/object/number serialization. Query names and values are encoded as URI query components; header strings ride under their declared names. This URI-component rule is an explicitly labeled OpenBindings convention needed to close the string-valued subset; it supplies no array or object style. A name collision with an address contribution or credential on the same query/header channel refuses before dispatch.
- **`encode`** supplies a JSON or text lane only where the selected input message has no effective content type; it cannot contradict or replace a declaration.
- **`websocketMessageType`** supplies exactly `text` or `binary` for a WebSocket publish. It is required because neither the artifact nor the incorporated binding chooses one; there is no array-order, payload-sniffing, or OpenBindings default. A `text` choice with encoded octets that are not valid UTF-8 refuses before the write.
- **`decode`** supplies a JSON or text lane only where the one governing output content type is absent; it cannot contradict a declaration or choose among several distinct declared types.

A supplied value for a server variable or address parameter honors the artifact's declared `enum`; an out-of-enum substitution is refused. A complete-URL or whole-address replacement is a distinct target override, not a claim that an out-of-enum substitution satisfies the declaration. Points also refuse a name the artifact does not declare, a missing required choice, and no resolvable server at all.

The **carriage** — the concrete value a consumer supplies to a point (its JSON shape in a JSON-configured runtime, or the idiomatic type in another) — is each implementation's own surface, not specification content: nothing an OBI carries depends on it, and pinning it would force every implementation's configuration API into one shape (core [§1.3](../../openbindings.md#13-authority-and-deferral), the completeness doctrine in the [catalog README](../README.md)). The reference SDKs document their shapes; a consumer's configuration is not portable across implementations by this specification's guarantee.

**Target URL assembly** (part of resolving the selected server): scheme from the resolved `protocol`; authority from the server's `host` with every variable substituted from consumer-supplied values, else the variable's declared `default` — an unsubstitutable variable is a pre-dispatch refusal; path = the server's `pathname` (if any) concatenated with the expanded address, normalized to exactly one `/` at the join. Each channel-parameter substitution is UTF-8 percent-encoded as one RFC 3986 path-segment value before replacement; this is an explicitly labeled OpenBindings convention because AsyncAPI identifies the value source but does not define a URI-template serialization. Concatenation, not RFC 3986 resolution: the pathname prefix is preserved.

### 9.3. Decode

Media types in this section are parsed under RFC 9110. Type, subtype, and parameter names compare case-insensitively; parameter ordering and quoted-string versus equivalent token spelling do not create distinct identities; parameter values compare under the parameter's governing semantics, and otherwise byte-for-byte after quoted-string unescaping. For an HTTP response, a declared effective content type matches the actual `Content-Type` when type and subtype agree and every declared parameter is present with the same value; additional actual parameters do not disqualify it. The matching declaration with the greatest number of parameters governs. Distinct declarations tying at that greatest specificity are ambiguous and produce a protocol error. This parameter-subset/specificity rule is an explicitly labeled OpenBindings convention: AsyncAPI supplies the media declarations and HTTP supplies their syntax, but neither defines how several parameterized reply-message declarations select one response. For a WebSocket subscription, which carries no per-message MIME field, “the same effective content type” below means the same complete parsed identity, parameters included.

The decode point applies **per declared delivery unit** — the HTTP response body of a declared reply, or each complete WebSocket data message of a subscription after RFC 6455 fragmentation reassembly — and output values are emitted as units arrive (**ASYNC-P-05**). A frame boundary never reaches this layer. The governing declarations are direction-correct:

- For an HTTP **publish**, only the operation's declared reply can govern output. A publish with no declared reply emits no value even if the peer sends a body: the transport body is not promoted into a message the artifact does not declare. When a reply is declared, messages whose HTTP message binding `statusCode` equals the final response status govern; when none match, messages without a declared `statusCode` govern. A non-empty body with no governing message in that declared reply contradicts the artifact and is a protocol error. If the governing candidates contain several distinct effective content types, the response's actual `Content-Type` MUST match exactly one of those declared types and narrows the governing set; an absent or unmatched header is a protocol error. If some governing candidates declare a type and others omit it, the set is likewise ambiguous and is refused unless the actual header selects one declared type. An empty response body emits no value.
- For a WebSocket **subscription**, the operation's resolved message list governs every complete data message. An absent or empty list makes the binding unresolvable. Because neither the AsyncAPI WebSockets binding nor the RFC 6455 text/binary opcode identifies an AsyncAPI Message Object or MIME type, every possible message MUST have the same effective content type, or every possible message MUST omit it. A mixture of absent and declared types or several distinct declared types is refused at binding resolution; `decode` never overwrites that ambiguity. A WebSocket publish with a declared reply is excluded at binding resolution by [§8](#8-target-and-interaction).

For each governing message, effective content type is its `contentType`, else the document's `defaultContentType`. After the direction-specific narrowing above, exactly one distinct effective type selects the lane: strict JSON for `application/json`/`+json` (malformed declared JSON is a loud error), text otherwise. JSON octets decode as UTF-8. Text-lane octets become strings per the declared charset, defaulting to UTF-8, and invalid sequences are a loud error; an RFC 6455 text message has already satisfied the protocol's UTF-8 requirement, while a binary message is decoded from its octets here. If every remaining governing message has **no** effective type, `decode` is required to supply the JSON or text lane. Invocation without that choice refuses before reading a delivery as an output. The point fills declaration absence only: the specification never converts conflicting declarations into configuration, a silent text preference, or payload sniffing.

### 9.4. Classification

Classification is **transport-native and not a configuration point** (**ASYNC-P-06**): this family declares no message-level failure semantics, so an invocation's outcomes are its transport's, per cell. AsyncAPI does not define redirect following, so whether to follow is runtime policy. For an HTTP publish, any followed redirect MUST preserve the artifact-declared method and complete encoded message body; a redirect whose semantics require, or whose configured client policy would perform, a method rewrite is not followed and remains the final response. For a WebSocket connection, any followed redirect MUST preserve a valid upgrade handshake and every credential/channel-collision floor of [§9.5](#95-credentials). A unary publish succeeds **iff** the final status after any such semantics-preserving redirects is 2xx. A client-streaming publish succeeds **iff** the closing handshake completes after the caller's clean input close — peer close codes and message content are never consulted. A subscription is established per its cell's rule ([§8](#8-target-and-interaction)) after any semantics-preserving redirects, completes on the transport's clean close, and otherwise fails as the transport fails or as a **declared implementation bound** is exceeded — and where implementation policy bounds a subscription's undelivered output, exceeding the bound fails that subscription **loudly**: output values are never silently discarded, the integrity floor this family adds. Message content never reclassifies an outcome (an `{error}` envelope is a successful delivery whose meaning is the consumer's business, unwrapped by the operation's output transform).

### 9.5. Credentials

Security metadata is never extracted into the OBI: the artifact's declarations are read at invocation time (**ASYNC-P-07**). The declaration semantics are the AsyncAPI specification's, incorporated — and they are **conjunctive**: the targeted server's `security` applies (the server the connection actually goes to, or, under a full-URL override, the server the default selection would have targeted), and the operation's `security`, when declared, applies **in addition**; within each declared list, satisfying one entry suffices. Each entry is a security-scheme object or `$ref` into `components.securitySchemes`.

What this specification pins is **wire application**: an `httpApiKey` scheme's credential rides its declared `in`/`name` (header, query, or cookie); `http`/`basic` and `http`/`bearer` — matched case-insensitively per RFC 9110 — ride the `Authorization` header per their RFCs; `oauth2` and `openIdConnect` access tokens ride `Authorization: Bearer`. For WebSocket connections, credentials ride the **upgrade request** — headers, or the upgrade URL's query string for schemes whose declared location is `query`. A scheme this list does not pin (`apiKey`, `scramSha256`, `X509`, an `http` scheme value other than `basic`/`bearer`, and any other type) is **surfaced to the consumer** rather than silently skipped — and so is a **pinned** scheme whose carriage the platform cannot apply (a header credential on an upgrade request the platform strips): surfaced, never silently dropped or rerouted. Alternate carriage — query tokens for header schemes, in-band auth messages — is consumer configuration, never this specification's coverage: **no credential ever rides a message body or a first WebSocket data message** under this document.

A credential and a declared channel-binding value can land on one channel: a websockets binding declares `query` and `headers` values that ride the same upgrade request a query- or header-riding credential rides. A name collision between them is **refused before dispatch** — loud, never a silent overwrite in either direction, and never a duplicated key (the `openbindings.openapi@1` sibling refuses the same species of collision at [its §9.6](../openapi/openbindings.openapi.md#96-security-declarations-credentials-and-channel-assembly)). HTTP header names compare case-insensitively under RFC 9110; query and cookie names compare exactly.

After one server-security alternative and, where declared, one operation-security alternative are selected, their credential destinations MUST likewise be pairwise non-colliding unless both requirements resolve to the same security-scheme object and one credential satisfies that same declaration. A header credential MUST NOT replace a processor-owned transport field: `host`, `content-length`, or `content-type` for an HTTP publish; and `host`, `connection`, `upgrade`, `sec-websocket-key`, `sec-websocket-version`, or `sec-websocket-protocol` for a WebSocket upgrade. A colliding selection is unusable and refuses before dispatch unless another complete artifact-declared selection is available. This is refusal of contradictory artifact channels, not a new precedence rule.

Where an implementation pools or reuses connections, a pooled connection MUST NOT be shared across differing credential identities; pooling itself remains implementation policy. How a consumer acquires credentials is a runtime concern; the project's [`binding-invoker`](https://openbindings.com/interfaces/binding-invoker) interface defines one such negotiation surface (informative).

### 9.6. Transform positions

This specification defines **no** context bindings at transform positions: a transform on an asyncapi binding evaluates in the core's closed environment, unaugmented.

## 10. Conformance

Rules carry stable identifiers under the same discipline as the core's: never reused, never renumbered. Source rules bind OBI content governed by this specification; processor rules bind implementations claiming support for `openbindings.asyncapi@1`. Verification follows the core's partial-verification posture.

- **ASYNC-D-01**: `content`, when present, is the parsed AsyncAPI document object or its source text as a string under [§3](#3-accepted-source-representations)'s grammar pin.
- **ASYNC-D-02**: `location`, when present, is an absolute URI addressing the AsyncAPI document, per [§4](#4-location).
- **ASYNC-D-03**: `ref` is present and is a JSON Pointer `#/operations/<operation-key>` addressing an operations-map entry — the only conformant spelling, RFC 6901-escaped, never percent-decoded, Reference Objects resolved — per [§7](#7-ref).
- **ASYNC-P-01**: The exact AsyncAPI 3.0.0 acceptance pin, YAML 1.2.2 string grammar, and loud refusal discriminate on the artifact's `asyncapi` field per [§2](#2-scope-and-incorporated-authorities) and [§3](#3-accepted-source-representations).
- **ASYNC-P-02**: The artifact's application perspective is incorporated and the invoker acts as counterparty only for the interaction cells [§8](#8-target-and-interaction) defines: `receive` publishes over HTTP or WebSocket; `send` subscribes over WebSocket; standalone `send` over HTTP is refused before dispatch rather than interpreted as SSE; a reply-bearing WebSocket `receive` is refused rather than having its declared replies discarded. A supported HTTP `receive` requires the artifact's HTTP operation binding to declare its method; POST is never inferred. The selected artifact server's protocol fixes the cell, and a target override with a different scheme refuses. Declared protocol `bindings` and their incorporated revision defaults are authoritative where they speak, explicit out-of-revision binding versions are refused, and omission is frozen to this identifier's incorporated revisions rather than mutable “latest” behavior.
- **ASYNC-P-03**: The input value is the whole message payload; one artifact-declared operation message governs it, with required `message` selection when several remain; encoding follows that message's effective content type and requires an `encode` lane when none is declared; each supported cell's presence rule holds, all with [§9.1](#91-input-mapping-and-encoding)'s refusals. A selected input, output, or reply message declaring Message Object `headers` makes the binding unresolvable in revision 1; those headers are never discarded or folded into the payload. WebSocket delivery units are complete RFC 6455 data messages after fragmentation reassembly, never frames.
- **ASYNC-P-04**: The server, address, message, protocolFields, encode, websocketMessageType, and decode configuration points behave per [§9.2](#92-configuration-points) — sole artifact choices select themselves while unresolved alternatives or missing required values refuse; URL replacement retains the selected server's protocol and artifact semantics; supplied-else-default variable substitution honors declared enums and refuses undeclared names; address-parameter `location` expressions remain authoritative and are evaluated only where the outgoing message exists before connection, while unsupported pre-connection expressions refuse; protocol-field schemas receive only declared string values and never JSON Schema defaults or invented non-string serialization; URL assembly concatenates and encodes substitutions as defined; a WebSocket publish requires an explicit text/binary choice because the artifact does not make one. The value-carriage a consumer uses to supply a point is implementation surface, not this rule's content.
- **ASYNC-P-05**: Decode applies per declared delivery unit with direction-correct governing declarations — an HTTP response produces output only for an artifact-declared reply message selected by its HTTP `statusCode` or status-unspecified fallback and, where necessary, its actual `Content-Type` under the parameter-aware most-specific rule; equal-specificity ties are protocol errors. Complete WebSocket subscription data messages use an operation message set with one shared complete parsed content-type identity or universal absence. `decode` fills only universal content-type absence and never chooses among conflicting declarations, per [§9.3](#93-decode).
- **ASYNC-P-06**: Classification is transport-native per [§9.4](#94-classification)'s per-cell rules, final-status on the unary cell, never message-content-driven, never configurable; redirect following is runtime policy but cannot rewrite an HTTP publish method, discard its body, or weaken a WebSocket upgrade's declared channel/credential rules; bounded subscriptions fail loudly, never silently dropping values.
- **ASYNC-P-07**: No security metadata is derived or written into OBI documents; requirement derivation is conjunctive per [§9.5](#95-credentials); wire application follows its pins; no credential rides in-band; unpinned and unapplyable schemes are surfaced; credential/channel-binding, distinct credential-destination, and credential/processor-owned-field collisions refuse before dispatch; pooled connections never span credential identities.

Conformance fixtures keyed to these identifiers are published with the project's conformance corpus. Deterministic _generation_ of OBI documents from AsyncAPI artifacts is a synthesis concern outside this specification; the project's interface-synthesizer and reference-tool documentation record those conventions.
