# `openbindings.openapi` Binding Specification

## 1. Status and identifier

This document is the normative text for the binding specification identifier **`openbindings.openapi@1`**, published by the openbindings project as its defining authority. The identifier is exact and opaque per core [OBI-B-01](../../openbindings.md#104-binding-specification-rules): no range or normalization semantics attach to it. An incompatible change to this specification publishes `openbindings.openapi@2` ([OBI-B-03](../../openbindings.md#104-binding-specification-rules)); compatible clarification may revise this document in place. It publishes with the OpenBindings core 0.2.0 change set, and reference tooling adopts the identifier — replacing the pre-bindingSpec `openapi@…` tokens — in the same coordinated change.

The key words "MUST", "MUST NOT", "REQUIRED", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [BCP 14](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174) when, and only when, they appear in all capitals.

## 2. Scope and incorporated authorities

This is the **openbindings project's** binding specification for [OpenAPI](https://spec.openapis.org/) documents. It is published under this project's authority, not by or for the OpenAPI Initiative: the OpenAPI Specification (OAS) is incorporated by reference and remains authoritative over the artifact and the HTTP mechanics it describes — document structure and reference resolution, parameter locations and serialization, request/response media declarations and encodings, server objects and variables, security scheme declarations, and everything else the artifact says about the wire. Plain HTTP semantics are [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110)'s; `text/event-stream` framing follows the WHATWG HTML server-sent-events processing model at the immutable source snapshot named below. This specification defines only the OpenBindings overlay: how a source carries or addresses an OpenAPI document, how a binding selects an operation, and how caller-facing operation values correspond to an HTTP exchange.

Like the core specification, this document defines portable meaning rather than an invocation API. Its HTTP-exchange, refusal, and ordering language states what acting on the binding denotes and which effects remain conformant; request objects, context negotiation, retries, cancellation APIs, and client architecture remain runtime or SDK surface.

**Normative upstream snapshot.** Revision 1 accepts only the published OAS editions [3.0.0](https://spec.openapis.org/oas/v3.0.0.html), [3.0.1](https://spec.openapis.org/oas/v3.0.1.html), [3.0.2](https://spec.openapis.org/oas/v3.0.2.html), [3.0.3](https://spec.openapis.org/oas/v3.0.3.html), [3.0.4](https://spec.openapis.org/oas/v3.0.4.html), [3.1.0](https://spec.openapis.org/oas/v3.1.0.html), [3.1.1](https://spec.openapis.org/oas/v3.1.1.html), and [3.1.2](https://spec.openapis.org/oas/v3.1.2.html), each interpreted under its own immutable official text. The string carriage grammar is [YAML 1.2.2](https://yaml.org/spec/1.2.2/). The SSE authority is the WHATWG HTML source at commit [`24c5e48bf66ea61bc199ec6338c81258275ba9c6`](https://github.com/whatwg/html/blob/24c5e48bf66ea61bc199ec6338c81258275ba9c6/source); the living rendered page and later upstream revisions are informative and do not alter this identifier. A later OAS patch or minor is never admitted merely because it matches a supported line; accepting another edition or advancing a normative authority pin publishes a new binding-specification identifier.

OpenAPI artifacts are substantially complete as actionable descriptions — routing, media types, and status declarations are native — so this specification's interpretation points ([§9.3](#93-servers-and-the-target-url), [§9.4](#94-configuration-points)) are fewer and narrower than an incomplete family's (compare [`openbindings.usage@1`](../usage/openbindings.usage.md)).

## 3. Accepted source representations

This specification accepts the **exact OpenAPI editions listed in §2** — 3.0.0 through 3.0.4 and 3.1.0 through 3.1.2 — in two carried representations:

- **Object** `content`: the parsed OpenAPI document as a JSON object.
- **String** `content`: the document's source text. This specification pins the string grammar — its own pin, not an OAS mandate: string content parses as **YAML 1.2.2** under the OAS's JSON-compatibility recommendations, of which JSON is a valid subset, so one grammar covers both spellings deterministically. Duplicate mapping keys are refused loudly.

Discrimination is the artifact's own **`openapi` field**: one of the exact accepted values is interpreted under that edition's OAS text; every other value (including a future 3.0 or 3.1 patch, OpenAPI 3.2, and Swagger 2.0's `swagger` field) is refused loudly at load (**OAPI-P-01**). No other representation is defined; there is no non-textual artifact form.

## 4. `location`

A source's `location`, when present, is an **absolute URI addressing the OpenAPI document itself** (**OAPI-D-02**) — `https://example.com/openapi.json`, `file:///srv/api/openapi.yaml`. Dereferencing it yields an accepted representation ([§3](#3-accepted-source-representations)). A bare filesystem path is a relative reference in form and is not a conformant `location` (core [OBI-D-05](../../openbindings.md#102-document-rules)).

## 5. `content`

A source's `content`, when present, MUST be one of the two accepted representations of [§3](#3-accepted-source-representations): the parsed document object, or its source text as a string (**OAPI-D-01**). No other JSON type is an accepted `content` value under this specification.

## 6. Composition

When `content` is present it is the artifact the processor interprets, per the core's content-primacy floor ([§5.4](../../openbindings.md#54-sources)).

A co-present `location` is the document's origin, and it serves as the embedded artifact's **base URI**: references internal to the embedded document (relative `$ref`s, relative server URLs) resolve against it per the OAS's own reference-resolution rules, exactly as they would had the document been retrieved from that address. The base travels in the OBI, so resolution is identical everywhere the OBI goes; the OBI's own retrieval URI is never a base (core [§7](../../openbindings.md#7-reference-resolution)). Embedded content with **no** co-present `location` has no base and MUST be self-contained: every internal reference resolves without one (bundle before embedding).

Self-containment is RECOMMENDED even when a `location` is present: a document whose embedded artifact leans on its base for `$ref` resolution requires that address to be reachable to interpret, which reintroduces the network dependence embedding exists to remove.

For a `location`-only source, the document's base URI is the `location`, per the OAS.

## 7. `ref`

A binding's `ref` is REQUIRED — the OAS defines no whole-artifact invocation — and MUST be a JSON Pointer of the form `#/paths/<escaped-path>/<method>`, addressing an **operation object**: the path segment carries [RFC 6901](https://www.rfc-editor.org/rfc/rfc6901) escaping (`/` → `~1`), and `<method>` is one of the OAS's HTTP method keys, **lowercase exactly as the artifact spells them** (**OAPI-D-03**). `#/paths/~1tasks/post` addresses the POST operation on `/tasks`.

The pointer is a **verbatim string**. Its `#/` shape is JSON Pointer notation, not an invitation to URI processing: no percent-decoding is ever applied, at authoring or at resolution. Each addressable operation has exactly one conformant spelling — the RFC 6901-escaped form above — and a `ref` matches it byte-for-byte, extending the core specification's literal-form doctrine (core [§7](../../openbindings.md#7-reference-resolution)) to this family's selector. A percent-encoded rendering of that spelling (`#/paths/~1users~1%7BuserId%7D/get` for `#/paths/~1users~1{userId}/get`) is not the spelling and denotes nothing.

Pointer evaluation follows **OAS reference resolution**, not raw JSON traversal: a path item that is a `$ref` (including a 3.1 `components.pathItems` target) is resolved before the method segment evaluates, so bundled artifacts using path-item references stay addressable. A well-formed `ref` whose path or method is absent from the resolved artifact makes the binding unresolvable; verification requires the artifact, per the core's partial-verification posture.

Only `paths` operations are addressable. **Webhooks and callbacks are excluded** from revision 1: they describe inbound calls the service makes to the consumer, an interaction this revision does not bind. The exclusion is a definition, not an open item.

## 8. Target and interaction

The binding denotes the addressed operation as **one HTTP exchange** against the resolved server ([§9.3](#93-servers-and-the-target-url)).

**Success classification and response declaration lookup are separate.** Classification is the final-status 2xx rule of [§9.4](#94-configuration-points). After a response arrives, its governing OAS Response Object is selected exactly as the OAS Responses Object declares: the exact status-code entry first, then the matching `2XX` range entry, then `default` when neither more-specific entry exists (**OAPI-P-06**). Thus `default` can govern the shape and media of a successful 2xx response; it does not itself make a non-2xx response successful. A **success response declaration** below means any entry that can govern at least one 2xx status under that lookup: an exact 2xx entry, `2XX`, and — unless `2XX` covers the whole class — `default`.

The interaction shape is bounded by declaration and selected by framing (**OAPI-P-06**):

- An operation is **streaming-capable** iff `text/event-stream` appears among the declared media types of at least one success response declaration. Capability is static: a consumer knows from the artifact alone whether streaming can occur.
- For an actual successful response, the exact status first selects its governing Response Object by the lookup above. Only that object's `content` map can govern framing. A concrete `text/event-stream` declaration there yields a **server-streaming** interaction; any other concrete declared media type yields a **unary** interaction. A media type declared only by a different status, range, or shadowed `default` entry is irrelevant to this response.
- A non-empty successful response with no governing Response Object, or whose `Content-Type` does not match a concrete media type in that object's `content` map, contradicts the artifact and is a protocol error. An empty body needs no media declaration and is unary. Thus a `text/event-stream` response is never accepted merely because SSE appears somewhere else in the operation.

A **unary** interaction carries zero or one caller input value and **zero or one output values**: a non-empty successful response emits one decoded value, while an empty successful response completes without emitting a value. Absence of a representation is not the JSON value `null`; a non-empty JSON body whose bytes spell `null` emits that explicit value. A **server-streaming** interaction likewise carries zero or one caller input value; each received SSE event emits one output value per the WHATWG processing model: the event's `data` lines joined with U+000A form the event's text, comment-only and empty-`data` events emit no value, and the `event`, `id`, and `retry` fields **never enter the output value** — whether an implementation surfaces them out of band is its own concern. Stream end completes the interaction; abnormal termination — transport failure before stream end — is a failure outcome, and output values already emitted **stand**. NDJSON and other streaming framings are excluded from revision 1.

## 9. Operation-boundary correspondence

### 9.1. Input mapping — the flattened model

The caller-facing input value is one JSON object (or absent). It is a **candidate-specific flattened view** of the operation's declared surface: parameters from every location and the request-body schema belonging to the selected Media Type Object merged into one object. An OAS Request Body Object may declare a different schema for every media type; there is no synthetic operation-wide “body schema.” Processors evaluate each concrete revision-1 media candidate independently, using that candidate's own resolved schema, and `requestMedia` selects among the candidates whose flattened view can carry the invocation. Fields map by name (**OAPI-P-03**):

- The declared parameter set is the merge of path-level and operation-level `parameters`, operation-level winning on same name-and-location collision (per the OAS), after applying the OAS's own ignored-header rule: a `header` parameter named `Accept`, `Content-Type`, or `Authorization` is ignored and contributes no input field, requirement, or wire value. Those names compare case-insensitively as HTTP field names. Media negotiation and security declarations remain their governing authorities; a parameter object cannot replace them.
- `path` parameters substitute into the URL template; `query` parameters join the query string; `header` parameters ride as request headers; `cookie` parameters join the `Cookie` header ([§9.6](#96-security-declarations-credentials-and-channel-assembly) defines channel assembly).
- **Serialization follows the OAS's `style`/`explode`/`allowReserved` rules, incorporated wholesale** — including their per-location defaults and their array and object expansions (**OAPI-P-02**). A `content`-form parameter (schema-less, per the OAS's parameter `content` map) serializes its value per its declared media type and rides its location as that serialized string. This specification adds no serialization of its own.
- Two parameter declarations sharing one name across **different locations** are legal under the OAS's name-plus-location identity but cannot be represented by this flattened value. That collision makes the operation unresolvable under revision 1. Effective header parameters whose names differ only by ASCII case likewise make it unresolvable: OAS parameter names are case-sensitive while HTTP field names are not, and revision 1 does not choose overwrite, comma-join, or repeated-field semantics for two artifact-distinct values. A parameter/body-property collision is evaluated per request-media candidate because body properties belong to that candidate's schema: it makes that candidate inadmissible, not a different non-colliding artifact alternative inadmissible. If every otherwise supported candidate is inadmissible, the binding is unresolvable. OpenBindings does not infer that independently declared wire values are equal, duplicate one caller value into both locations, or choose one declaration over the other. A future binding-specification revision may define a disambiguated boundary; revision 1 narrows coverage instead of changing the artifact's meaning.
- A field matching no declared parameter or property of the candidate's body schema **passes through into the body** only where that request-media candidate gives it a destination **without invention** — the **JSON family** (`application/json`, any `+json` suffix), where an extra field is simply another body property. In every other case a field matching neither a declared parameter nor a candidate body property makes that candidate inadmissible — loudly, never silently dropped: when no request body is declared; when the candidate's body is **non-object** (the flattened contract carries the synthetic `body` property of the next rule — a string, binary, array, any non-object shape), so there is no property surface to join; and for `multipart/form-data` or `application/x-www-form-urlencoded`, whose OAS serialization derives each part's carriage from per-property declarations an unmatched field lacks (the same absence the degenerate media/schema refusal turns on, [§9.2](#92-request-and-response-media)). These are the same species of refusal as the unflattenable collisions above and the missing-required-path-parameter rule below. The rule is stated as a property, not a format allowance: the operation value domain is JSON (core [§5](../../openbindings.md#5-document-model)), so a JSON body needs no translation step for an extra field, and a future request media whose serialization is derivable without a per-property declaration would qualify by the same test. Where passthrough holds, the field joins that candidate's body value: it rides the JSON body's encoding, never a side channel. No schema evaluation participates in this routing; enforcing the body schema is the server's business.
- A non-object schema in a request-media candidate is represented in that candidate's flattened contract under the synthetic **`body`** property; at the wire, that property's value **is** the request body, unwrapped. Object-or-not is decided from the **resolved candidate declaration**, never from a caller value. Reference Objects and Schema Object `$ref` keywords are resolved under the artifact's own edition before this test. Direct `properties` and an explicit `object` type make the declaration object-shaped. An `allOf` is object-shaped when at least one recursively resolved member is object-shaped; its flattened property surface is the union of directly declared properties and the recursively collected properties of all members. Resolution visits each referenced schema identity at most once, so a recursive schema terminates without inventing additional properties. If any schema examined on that top-level reference/`allOf` traversal declares `oneOf`, `anyOf`, `not`, `if`, `then`, or `else`, that candidate is inadmissible in revision 1; this revision does not assign conditional schemas a union, preference, or value-dependent route. A **typeless** schema after those resolutions (a bare `{}` or description-only schema) is non-object and rides the synthetic `body` property exactly as an array or scalar does, whatever values it might admit at runtime.

**Required declarations.** An invocation supplying **no input value** while the operation declares any required parameter or a required request body is refused before dispatch. A **supplied** input object missing required members is sent as-is — the server's declared validation is the authority — with one exception: a missing declared **path** parameter always refuses before dispatch, because the URL cannot be built without it.

**The remaining body and candidate election.** For each declared concrete revision-1 media type, the processor resolves that Media Type Object's schema and applies the rules above to determine whether the candidate can carry this invocation. When a JSON-family candidate is selected and every input field was consumed by parameters, the body is `{}` if the artifact declares the request body required, and omitted otherwise. `requestMedia` then selects one admissible candidate as [§9.2](#92-request-and-response-media) defines. Request-media selection is consulted only when the invocation will emit a body: a required request body, a supplied synthetic `body`, or at least one supplied body-property/passthrough field. An invocation that emits no body does not consult `requestMedia`, does not require a candidate, and omits both the body and `Content-Type`. This is what permits an ordinary parameter-only operation with no `requestBody`, and an invocation omitting an optional body, to dispatch without inventing a media type.

### 9.2. Request and response media

Media types throughout this section are parsed under RFC 9110. Type, subtype, and parameter names compare case-insensitively; parameter ordering and quoted-string versus equivalent token spelling do not create distinct identities; parameter values compare under the parameter's governing media-type semantics, and otherwise byte-for-byte after quoted-string unescaping. Two keys in one OAS `content` map that denote the same parsed media type are a **normalized collision** and make that request or response candidate set unresolvable — no first-key rule is invented.

Parameters are never discarded. For **request selection**, each semantically distinct concrete content-map key is its own artifact-declared candidate; a configured `requestMedia` value selects by parsed media-type identity, and the emitted `Content-Type` carries that declaration's parameters. For **response matching**, a concrete declaration matches the actual `Content-Type` when type and subtype agree and every parameter present in the declaration is present with the same value in the actual field; additional actual parameters do not disqualify it. Of the matching declarations, the one with the greatest number of declared parameters governs. If several distinct declarations tie at that greatest specificity, the response is ambiguous and is a protocol error rather than an array-order preference. This most-specific-subset rule is an explicitly labeled OpenBindings convention: OAS permits parameterized content keys and HTTP permits additional response parameters, but neither supplies the selection algorithm needed at this boundary. It preserves a general `application/json` declaration alongside a more specific profiled declaration without collapsing them.

Media ranges (`*/*`, `application/*`) never participate in revision-1 selection. A range remains an upstream-valid declaration, but this revision does not have a concrete boundary encoding to select from it: a body-emitting request with no concrete candidate refuses before dispatch, and a non-empty response governed only by ranges is a protocol error. The range is never silently treated as absent or as an exact media type.

When the invocation will emit a body, its media type is selected from the operation's **declared** `requestBody.content` under the **`requestMedia`** configuration point (**OAPI-P-04**, [§9.4](#94-configuration-points)). Revision 1 can carry exact `application/json`, `+json` types, `multipart/form-data`, `application/x-www-form-urlencoded` (fields serialize per the OAS's `encoding` rules), and `text/plain` for a string body. The artifact's content map declares alternatives, not a preference order: consumer configuration may choose any candidate admissible under its own Media Type Object schema and [§9.1](#91-input-mapping--the-flattened-model), and without a configured choice a processor may choose any admissible candidate. Different choices inside that set are conformant. A configured media type that is undeclared, unsupported by revision 1, or inadmissible for the invocation is refused before dispatch; configuration never substitutes another candidate's schema. A body-emitting invocation with no admissible candidate is likewise refused. A no-body invocation bypasses this point as defined in [§9.1](#91-input-mapping--the-flattened-model). Non-JSON _response_ bodies are not excluded; they decode through the text lane of [§9.4](#94-configuration-points) only after the governing Response Object declares their media type.

**Degenerate media/schema combinations.** `multipart/form-data` or `application/x-www-form-urlencoded` is not a candidate when its own declared schema does **not** flatten under [§9.1](#91-input-mapping--the-flattened-model)'s declaration-only determination (no `properties` and no explicit `object` type: the shapes the flattened contract carries under the synthetic `body` property); `text/plain` is a candidate only for a non-flattening declaration and a supplied string value. The OAS specifies multipart and form-urlencoded serialization exclusively over object properties: its multipart considerations define the payload as an object whose properties become the parts, and its Encoding Object is keyed by property name — for a propertyless shape no rule exists to implement, and anything emitted would be implementation invention. The text lane is the inverse: a single-string wire that cannot carry a flattening schema's object contract. Candidate exclusion is decided by that candidate's declarations (and, for text, the value's required string type), never by payload sniffing. A future revision may define carriage for another combination if the OAS or a real artifact supplies a serialization rule.

**Multipart parts.** A part is binary-signaled per the artifact's edition, following the OAS's own binary-data model: 3.0.x by `format: binary`; 3.1.x by a schema carrying `contentMediaType` **with `type` omitted** (the OAS's raw-binary shape, OAS 3.1 §"Working with Binary Data"), or `type: string` **with `contentEncoding`** (its encoded shape). A binary-signaled part's bytes come from the caller's string value: decoded per the schema's declared `contentEncoding` where one is declared (the 3.1 encoded shape), and by **Base64** where the artifact signals binary without declaring an encoding (3.0.x `format: binary`, or the 3.1 raw shape) — this specification's boundary encoding for bytes, since the operation value domain is JSON (core [§5](../../openbindings.md#5-document-model)). Parts that are not binary-signaled serialize per the artifact's `encoding` object where present, else per the OAS's per-type part defaults. Nothing here is decided by the value's bytes; the artifact's declarations decide.

The request `Accept` header advertises the declared **concrete** media types of the operation's success response declarations ([§8](#8-target-and-interaction)). Membership and artifact-declared parameters are normative; ordering and quality preferences are not. When that set is empty, the header is omitted unless consumer configuration supplies one — this specification does not invent `application/json` for an artifact that declared no response media.

### 9.3. Servers and the target URL

The target URL is the resolved server joined with the operation's path template. Server resolution is a **named configuration point** (**OAPI-P-05**) with the effective-choice semantics below:

- The **effective server list** is the OAS's: the operation's `servers`, else the path item's, else the document's, else the OAS-defined implied server of `url: "/"`.
- An effective list with exactly one entry selects that entry. With more than one, the artifact has declared alternatives without a preference and the **server** point is required: consumer configuration selects a member, or invocation refuses before dispatch. Consumer configuration may supply that member's server-variable values or replace its resolved base URL outright; a URL replacement changes the target, not which artifact server's declarations and security semantics govern.
- A supplied server-variable value honors the artifact's declared `enum`; a value outside it is refused. A complete-URL replacement is a distinct target override, not a claim that an out-of-enum substitution satisfies the Server Variable Object. The concrete carriage of a supplied override is implementation surface, not specification content; the reference SDKs document their shapes.

A relative effective-server URL (the implied `/` included) resolves against the artifact's base URI ([§6](#6-composition)) per RFC 3986. A missing selection from a multi-entry list, an out-of-enum variable value, or a server URL that cannot resolve to an absolute URL — the implied `/` with no base URI, for instance — is a pre-dispatch refusal.

### 9.4. Configuration points

This family has **four** named configuration points — `server` resolution ([§9.3](#93-servers-and-the-target-url)), `requestMedia` ([§9.2](#92-request-and-response-media)), and the two interaction points below. Each row defines the meaning of one **effective** choice and the artifact answer or binding convention when none is supplied. How a runtime chooses among per-call, standing, or other configuration sources is outside this specification; those scopes cannot change the point's semantic effect. All binding conventions are content-independent: decided by declarations and framing, never payload bytes.

| Point              | Default                                                                                     | Meaning                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ------------------ | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **`requestMedia`** | no preference within the declared candidates of §9.2; not consulted when no body is emitted | Which artifact-declared request media type carries the body. A configured exact media type selects one candidate; without configuration a processor may choose any candidate. It cannot cause an otherwise bodyless invocation to emit a body.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **decode**         | **the governing-response header rule** (**OAPI-P-07**)                                      | An empty body (204 included) completes without an output value and without consulting media. For a non-empty body, the final status selects the governing Response Object per [§8](#8-target-and-interaction), and the response's `Content-Type` MUST select exactly one concrete entry in that object's `content` map under §9.2's parameter-aware most-specific rule; an absent, unmatched, or ambiguous header is a protocol error. A matched `application/json` or `+json` type parses as strict JSON — malformed declared JSON is a loud error, never a silent string. Every other matched concrete type is the **text lane**: bytes become a string per the header's `charset` parameter, defaulting to UTF-8, and invalid sequences are a loud decode error (a consumer needing raw or non-text bytes overrides at this point; revision 1 defines no base64 response lane). For a server-streaming interaction the matched response media is `text/event-stream`; the point applies **per event** to the event's U+000A-joined data text ([§8](#8-target-and-interaction)), whose default output is that text — SSE carries no per-event media declaration, so JSON-emitting event data requires consumer configuration at this point. |
| **classify**       | **the 2xx rule** (**OAPI-P-08**)                                                            | Success **iff** the final status ∈ 2xx. Declared `responses` refine what a failure _means_, never whether it is one — a declared 404 documents a failure's shape, it does not bless the failure. OpenAPI does not define redirect following, so whether to follow is runtime policy; any followed redirect MUST preserve the bound operation's HTTP method and complete request body. A redirect whose semantics require, or whose configured client policy would perform, a method rewrite is not followed and remains the final response. Classification applies to the **final** status after any semantics-preserving redirects, and a surviving 3xx is not a success. Limits and cross-origin policy remain runtime concerns. |

Field routing is not a configuration point in this family: the artifact answers routing ([§9.1](#91-input-mapping--the-flattened-model)).

### 9.5. Success and the output value

Which outcomes are successes is decided by the **classify** point. For a successful non-empty unary response, the operation's one output value is the product of the **decode** point; an empty successful response emits none. Each streamed output value is the per-event product of that point. Response headers have **no representation in the output value** in revision 1 — a declared exclusion. Failure outcomes are not operation results and have no representation in this specification; what a consumer surfaces about them — and any error-code vocabulary — is its own concern, outside this document.

### 9.6. Security declarations, credentials, and channel assembly

Security metadata is never extracted into the OBI: the artifact's declarations remain authoritative whenever the binding is acted upon. The declaration semantics are the OAS's, incorporated: the operation's `security` **replaces** the document's entirely (an explicit `[]` means anonymous), requirements are an OR of ANDed schemes, and each scheme's definition lives in `components.securitySchemes`.

When more than one Security Requirement Object is satisfiable, **any one complete alternative** may be selected; the array declares alternatives, not an OpenBindings preference order. Consumer credential policy may choose among them. A processor MUST NOT construct apparent satisfaction by combining incomplete fragments from different alternatives. Different selections that each completely satisfy an artifact-declared alternative remain conformant behavior, not a binding-specification ambiguity.

What this specification pins is **wire application** (**OAPI-P-09**): an `apiKey` scheme's credential rides its declared `in`/`name` (header, query, or cookie); `http` schemes with values `basic` and `bearer` — matched case-insensitively, per RFC 9110's auth-scheme rules — ride the `Authorization` header per their RFCs; `oauth2` and `openIdConnect` access tokens ride `Authorization: Bearer`. A scheme, or an `http` scheme value, that this list does not pin (`http`/`digest`, `mutualTLS`, and any future type) is **surfaced to the consumer** rather than silently skipped.

**Channel assembly** (**OAPI-P-10**): declared cookie parameters and cookie-riding credentials merge into the single `Cookie` header — parameters in declaration order, credentials appended after. Within the selected complete security alternative, credential destinations MUST be pairwise non-colliding, and no credential destination may collide with an effective declared parameter at the same location: these are distinct artifact declarations even when the caller omitted the parameter, so one wire value never silently satisfies both. A header credential also MUST NOT collide with a processor-owned request field (`host`, `content-length`, `content-type`, or `accept`). A raw header declaration named `Cookie` is admissible by itself, but cannot coexist with any effective cookie parameter or selected cookie-riding credential: revision 1 does not parse a raw cookie string and assign precedence against structured cookie contributions. Such a security alternative is unusable and is refused before dispatch unless another complete, non-colliding artifact-declared alternative is selected. Header names compare ASCII case-insensitively; query and cookie names compare exactly. This catches two ANDed schemes that both require `Authorization`, two API keys naming one destination, an API key shadowing a parameter, and an API key attempting to replace media framing. No source silently overwrites, duplicates, or supplies another.

The same ownership rule applies before credentials: an effective header parameter targeting `host` or `content-length` makes the operation unresolvable. An effective raw `Cookie` header parameter is unresolvable when any effective cookie parameter also exists, and it makes a selected security alternative unusable when that alternative contributes a cookie credential. (`content-type` and `accept`, plus `authorization`, have already been removed by the incorporated OAS ignored-header rule.) A header parameter cannot replace target resolution, body framing, or structured cookie assembly merely because it was declared as an ordinary parameter.

How missing credentials are discovered, requested, and resolved is a consumer and runtime concern, not this specification's: the project's [`binding-invoker`](https://openbindings.com/interfaces/binding-invoker) interface defines one such negotiation surface (informative).

### 9.7. Transform positions

This specification defines **no** context bindings at transform positions: a transform on an openapi binding evaluates in the core's closed environment, unaugmented.

## 10. Conformance

Rules carry stable identifiers under the same discipline as the core's: never reused, never renumbered. Source rules bind OBI content governed by this specification; processor rules test whether implementations preserve the specified meaning when acting on it, not whether they expose one invocation API. Verification follows the core's partial-verification posture.

- **OAPI-D-01**: `content`, when present, is the parsed OpenAPI document object or its source text as a string, per [§5](#5-content).
- **OAPI-D-02**: `location`, when present, is an absolute URI addressing the OpenAPI document, per [§4](#4-location).
- **OAPI-D-03**: `ref` is present, and is a JSON Pointer `#/paths/<escaped-path>/<lowercase-method>` addressing an operation object under OAS reference resolution, in the pointer's one verbatim spelling — never percent-decoded — per [§7](#7-ref).
- **OAPI-P-01**: The exact accepted OAS edition set, YAML 1.2.2 string-content grammar pin, edition discrimination, and loud refusal behave per [§2](#2-scope-and-incorporated-authorities) and [§3](#3-accepted-source-representations).
- **OAPI-P-02**: Parameter serialization follows the OAS `style`/`explode`/`allowReserved` rules and defaults, with `content`-form parameters serialized per their declared media type, per [§9.1](#91-input-mapping--the-flattened-model).
- **OAPI-P-03**: Input fields map per the candidate-specific flattened model — OAS parameter merge and ignored-header rules; binding-resolution refusal for same-named parameters from different locations and case-colliding effective header parameters; per-media-candidate exclusion for parameter/body-property collision; reference resolution and bounded `allOf` property collection before each candidate's object-shape determination; exclusion of composition that cannot yield one declaration-defined surface; evaluation-free passthrough into a candidate object body only for JSON-family media; candidate exclusion for an unmatched field where no destination is derivable without invention; required-declaration refusals including the missing-path-parameter case; candidate election; the remaining-body rule; and synthetic `body` unwrapping — per [§9.1](#91-input-mapping--the-flattened-model).
- **OAPI-P-04**: A no-body invocation bypasses request-media selection and emits neither body nor `Content-Type`; a body-emitting invocation preserves the artifact's declared concrete candidate set with no binding-specification preference, honors an exact `requestMedia` choice, and refuses when no candidate can carry the declaration/value. Multipart part encoding includes the Base64 boundary encoding for binary-signaled parts. `Accept` advertises the normative concrete membership set and is omitted when that set is empty rather than inventing a media type, per [§9.2](#92-request-and-response-media).
- **OAPI-P-05**: Server resolution uses the OAS effective server list: a sole member selects itself; multiple members require consumer selection; variable substitutions honor declared enums; a complete URL may replace the selected member's target without replacing its artifact semantics; unresolved or unselected targets refuse before dispatch, per [§9.3](#93-servers-and-the-target-url).
- **OAPI-P-06**: Response declaration lookup is exact status → matching range → `default`, independently of final-status 2xx classification; streaming capability is statically bounded by all success declarations, while an actual response's interaction and media are governed only by the Response Object selected for its final status. A non-empty body with no governing declaration or no matching concrete media type is a protocol error; SSE events are extracted per the WHATWG processing model, per [§8](#8-target-and-interaction).
- **OAPI-P-07**: The decode configuration point defaults to the governing-response header rule: empty bodies emit no value; non-empty bodies require one parameter-aware, most-specific concrete media match under OAPI-P-06 and then use the defined JSON or text lane, including the per-event text default for SSE, per [§9.2](#92-request-and-response-media) and [§9.4](#94-configuration-points). Normalized content-map collisions and equal-specificity response ties refuse rather than selecting by source order.
- **OAPI-P-08**: The classify configuration point defaults to the 2xx rule, applied to the final status; redirect following is runtime policy, but any followed redirect preserves the bound operation's method and complete body, and a method-rewriting redirect remains the final response, per [§9.4](#94-configuration-points).
- **OAPI-P-09**: Credentials apply to the wire per scheme as [§9.6](#96-security-declarations-credentials-and-channel-assembly) pins; any one complete artifact-declared security alternative may be selected, incomplete fragments from different alternatives never manufacture satisfaction; unpinned schemes are surfaced, never silently skipped; security metadata is never written into OBI documents.
- **OAPI-P-10**: Channel assembly merges non-colliding parameters and credentials deterministically; an effective header parameter cannot replace processor-owned target, length, or cookie assembly; pairwise credential collisions, credential/effective-parameter collisions, and header-credential/processor-owned-field collisions make the selected security alternative unusable and refuse before dispatch unless another complete declared alternative is selected, per [§9.6](#96-security-declarations-credentials-and-channel-assembly).

Conformance fixtures keyed to these identifiers are published with the project's conformance corpus. Deterministic _generation_ of OBI documents from OpenAPI artifacts (operation-key derivation, output-schema selection, schema translation) is a synthesis concern outside this specification; the project's interface-synthesizer and reference-tool documentation record those conventions.
