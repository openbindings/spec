# `openbindings.usage` Binding Specification

## 1. Status and identifier

This document is the normative text for the binding specification identifier **`openbindings.usage@1`**, published by the openbindings project as its defining authority. The identifier is exact and opaque per core [OBI-B-01](../../openbindings.md#104-binding-specification-rules): no range or normalization semantics attach to it. An incompatible change to this specification publishes `openbindings.usage@2` ([OBI-B-03](../../openbindings.md#104-binding-specification-rules)); compatible clarification may revise this document in place. It publishes with the OpenBindings core 0.2.0 change set, and reference tooling adopts the identifier — replacing the pre-bindingSpec `usage@…` tokens — in the same coordinated change.

The key words "MUST", "MUST NOT", "REQUIRED", "SHOULD", "SHOULD NOT", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [BCP 14](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174) when, and only when, they appear in all capitals.

## 2. Scope and incorporated authorities

This is the **openbindings project's** binding specification for [jdx usage](https://usage.jdx.dev) CLI descriptors. It is published under this project's authority, not by or for the usage project: the usage specification as released in immutable [`usage` v3.5.6 (`84719de261e2cbd2db83d149bb7511c29f95206a`)](https://github.com/jdx/usage/tree/84719de261e2cbd2db83d149bb7511c29f95206a) is incorporated by reference and remains authoritative over the descriptor format itself — its KDL document syntax, command trees, flags, arguments, aliases, the `min_usage_version` directive, and all descriptor semantics. Later usage releases do not silently change `openbindings.usage@1`; adopting new descriptor semantics requires an explicit compatible revision of this document or a new binding-specification identifier. This specification defines only the OpenBindings overlay: how a source carries or addresses a descriptor, how a binding selects a command, and how caller-facing operation values correspond to a command invocation.

A usage descriptor describes a human command surface and cannot declare output decoding, exit-code meaning, or a field's stdin routing. This specification does not author that missing half into artifacts or OBI documents: it defines content-independent defaults at **named configuration points** ([§9](#9-operation-boundary-correspondence)), so the gap is made up in consumer configuration — the completeness doctrine of the [catalog README](../README.md#the-completeness-spectrum-specification--configuration--complete-invocation). The artifact stays pristine; the OBI stays abstract.

## 3. Accepted source representations

This specification accepts one representation: **usage descriptor source text** — a KDL document per the usage specification — carried as string `content` ([§5](#5-content)) or addressed by `location` ([§4](#4-location)). There is no second representation, so no discrimination rule is needed, and no encoding for non-textual artifacts is defined (usage has none).

Revision 1 interprets every descriptor under the pinned **usage v3.5.6** grammar and semantics; processors do not choose a private 2.x/3.x interpretation under this identifier. A descriptor's `min_usage_version` directive, when present, is a load-time gate (**USAGE-P-01**): its SemVer value MUST be less than or equal to 3.5.6, otherwise the descriptor is refused before any use. An absent directive uses the pinned grammar without inventing a minimum. Advancing the interpreter snapshot requires a revision of this binding specification.

Two upstream composition features are **excluded** from revision 1 (**USAGE-P-07**). A descriptor containing an `include` directive is refused at load: usage resolves its `file` as an operating-system path relative to the parsing file, while an embedded or remotely addressed OpenBindings source has no portable filesystem-path base with the same meaning everywhere it travels. A descriptor containing a command `mount` is likewise refused at load: resolving it executes a shell-word command and merges newly emitted commands into the artifact, importing an executable discovery phase and a mutable command surface that revision 1 does not bind. These are explicit coverage limits, not invitations for an implementation to pick a base, working directory, shell behavior, or execution policy privately. Authors expand includes and mounts before embedding the resulting descriptor.

Usage's **configuration-file and external-parse lanes are likewise excluded** from revision 1 (**USAGE-P-08**). A descriptor is refused at load when it declares a `config` block; a `config_file` or `config_alias` directive; a field-level `config` key; or an argument `parse` command. Configuration-file lookup depends on filesystem location, `findup`, environment interpolation, file-format parsing, aliases, and the artifact's CLI → environment → config-file → default precedence. An argument `parse` command adds an external process whose execution authority and output semantics are not the target invocation itself. Revision 1 neither loads those files or runs that command nor silently treats the declarations as absent. This narrower accepted subset preserves ordinary field `env` and `default` declarations, whose values and precedence are completely defined below without a filesystem or auxiliary process. A later revision may add the omitted lanes only with explicit working-directory, acquisition, authorization, and value-carriage semantics.

## 4. `location`

A source's `location`, when present, is one of two absolute address forms (**USAGE-D-02**). Both are binding-specification-defined absolute addresses in the sense of core [OBI-D-05](../../openbindings.md#102-document-rules): neither takes a base, and each means the same thing wherever the document travels.

1. **Document address** — an absolute URI addressing the descriptor document itself: `https://example.com/tool/usage.kdl`, `file:///opt/tool/usage.kdl`. Dereferencing it yields the descriptor source text.
2. **Exec address** — the scheme-like prefix `exec:` followed by an argv vector: one command token and zero or more argument tokens, separated by single spaces (`exec:mytool usage`). The addressed artifact is the standard output of executing that vector **directly** — never through a shell. **The dereference succeeds only when the process exits 0**; any other termination — a non-zero exit, a signal — is a loud dereference failure, and the standard-output bytes are **not** an artifact (a generator that emits a partial descriptor and then fails must never be read as a smaller valid one). Standard error is diagnostic, never the artifact. Unlike the document-address forms, `exec:` has no incorporated address scheme to supply this success condition — it is minted by this specification, so this specification states it. Tokens carry no quoting mechanism; a command whose arguments contain spaces is outside this form, and its generated descriptor is embedded as `content` instead. This form exists for tools whose descriptors are generated live; it is an absolute address by construction.

A bare filesystem path (`./usage.kdl`, `/opt/tool/usage.kdl`) is a relative reference in form and is not a conformant `location` (core OBI-D-05); local descriptors ride `file://` URIs or embedded `content` — embedding is the recommended lane for emitted documents, keeping them self-contained.

**Security.** Dereferencing an exec address executes a document-supplied command. A processor MUST NOT dereference an exec address without explicit prior authorization from its operator or configuration for that command (**USAGE-P-02**); the default is refusal. This is the one normative security floor this specification adds; all other mitigation posture follows core [§9](../../openbindings.md#9-security-considerations).

## 5. `content`

A source's `content`, when present, MUST be a JSON string carrying the UTF-8 source text of a usage descriptor (**USAGE-D-01**). No other JSON type is an accepted `content` value under this specification.

## 6. Composition

When `content` is present it is the artifact the processor interprets, per the core's content-primacy floor ([§5.4](../../openbindings.md#54-sources)). A co-present `location` is the descriptor's provenance — its document address, or the exec recipe that regenerates it — and MAY be used to refresh or compare, never as a competing artifact. Because revision 1 refuses usage's external `include` and dynamic `mount` composition features ([§3](#3-accepted-source-representations)), every accepted descriptor is self-contained and this specification defines no reference-base role for `location` (OBI-B-02 item 4: the answer is _none_).

## 7. `ref`

A binding's `ref`, when present, MUST be a non-empty **command path**: the command names along the descriptor's command tree, separated by single spaces (`db migrate run`) (**USAGE-D-03**). Resolution walks the tree from the root command; each segment matches a command's declared name or any of its declared aliases, exactly and case-sensitively. The spelling in `ref` is also the spelling emitted into argv: an alias is an artifact-declared alternative, not a hint to rewrite the segment to the command's primary name. A flag an ancestor command declares `global` accumulates onto the resolved command's effective surface. The incorporated usage specification defines the `global` attribute but does not state its scope of application; as a last-resort **OpenBindings convention**, this specification pins the ancestor-chain reading — a `global` flag reaches the declaring command and its descendants — so that one descriptor yields one effective surface in every implementation. A future upstream answer supersedes this convention only through a revision of this binding specification, never silently.

The **root command** is addressed by omitting `ref`. An empty-string `ref` is not conformant — this specification gives each meaning one spelling. A `ref` that resolves to no command in the artifact makes the binding unresolvable; verifying resolution requires the artifact, and a validator without it leaves the check unverified per the core's partial-verification posture.

## 8. Target and interaction

The binding target is the descriptor's non-empty **`bin`** value plus the command selected by `ref`, executed as **one direct process spawn per invocation** — argv assembled per [§9](#9-operation-boundary-correspondence), never interpreted by a shell (**USAGE-P-03**). Usage defines `bin` as the binary name and `name` as a friendly name; consequently `name` is never a target fallback. A descriptor without a non-empty `bin` is unresolvable. An `exec:` source location addresses acquisition of the descriptor and likewise does not replace its invocation identity. Consumer configuration may replace the executable address through the `target` point, but it cannot make a descriptor lacking `bin` resolvable: core target identity remains derivable from the binding and artifact alone.

The interaction is **unary**: at most one caller-facing input value, exactly one successful output value. An invocation supplying no input value runs the selected command only when its artifact-declared required arguments and constraints permit that invocation; otherwise it refuses before spawn. The spawned process is bound to the invocation's lifetime: cancelling the invocation terminates the process.

Interactive commands, PTY surfaces, and streaming interactions are **excluded** from revision 1. The exclusion is a definition, not an open item: a descriptor cannot express them, and a future revision that adds them is an incompatible change (`openbindings.usage@2`).

## 9. Operation-boundary correspondence

### 9.1. Input mapping

The caller-facing input value is one JSON object (or absent). Its fields map by **canonical field identity** onto the resolved command's effective surface (**USAGE-P-04**). Usage declares CLI spellings but no JSON property identity, so this is an explicitly labeled **OpenBindings correspondence convention**: a flag's identity is its first declared long spelling with leading hyphens removed, or, when it has no long spelling, its first declared short spelling with the leading hyphen removed. Its other spellings remain CLI aliases but are not additional JSON fields. A positional's identity is its declared argument name. Two effective flags, or a flag and positional, that produce the same identity make the binding unresolvable; input-property order never disambiguates them. A field matching no canonical identity is refused before spawn.

The artifact governs arity and repetition:

- A no-argument flag accepts a Boolean. `true` emits the canonical flag spelling; `false` emits its declared `negate` spelling when present and otherwise emits nothing. Any other value is refused.
- A count flag accepts a non-negative integer _N_ and emits the canonical spelling _N_ times.
- An ordinary value-bearing flag accepts one value and emits the flag followed by one token. A repeatable flag accepts an array and emits the flag/value pair once per element. A flag with a variadic argument accepts an array and emits the flag once followed by its element tokens. A declaration that is both repeatable and variadic is refused in revision 1 because usage does not define which wire form OpenBindings should prefer.
- A non-variadic positional accepts one value; a variadic positional accepts an array whose elements occupy consecutive argv tokens. An array for a non-variadic surface, or a scalar for a multi-value surface, is refused. Declared minimum and maximum counts are enforced.
- Each string contributes itself as one token. A non-string value on an ordinary value-bearing flag or positional has no usage-defined token representation and therefore requires the per-field **`encode`** configuration point; without one it is refused before spawn. The encoder MUST produce exactly one string token for each scalar occurrence. For repeatable or variadic surfaces it applies independently to each array element, preserving the artifact's occurrence structure; it never spreads an object or array into new argv positions. `null` is a supplied value, never “unsupplied,” and likewise requires an encoder where the artifact admits a value.

Within revision 1's accepted subset, all usage-declared defaults, environment fallbacks, requirements, choices, conflicts, and overrides remain authoritative. For a field omitted from caller input, the applicable pinned precedence is an explicitly configured value under that field's declared `env` name, then its declared default; a required field with neither is refused before spawn. Configuration-file precedence does not disappear into this sentence: descriptors declaring that excluded lane have already been refused under USAGE-P-08. A processor does not substitute an undeclared environment name or read an undeclared ambient host environment. For a caller-supplied field the default argv route is the artifact's highest-precedence admissible carriage; the `route` point may instead select the artifact's declared environment variable under the limits below. In particular, when two supplied flags participate in an `overrides` relation, the invocation is refused before spawn: the input object has no occurrence order from which to derive the artifact's winning flag. OpenBindings does not invent one.

An artifact may augment a field's literal `choices` from `choices env="NAME"`. The values come from that exact name in the invocation's explicitly configured process environment and are split on commas or Unicode whitespace, with empty elements removed, exactly as the pinned usage implementation defines. They are unioned with literal choices without duplicates. If that name is absent, no dynamic choices are added; when the artifact supplied no literal choices, every value consequently fails its artifact constraint. A processor MUST NOT consult its own ambient host environment as a private fallback: two invocations with the same artifact and declared environment have the same accepted set.

Argv order is independent of JSON object-property order. The incorporated descriptor declares surfaces but no semantic ordering for reconstructing one invocation from an unordered JSON object. As an explicitly labeled last-resort **OpenBindings convention**, the vector is assembled as: the executable selected by `target`; the `ref` segments in their exact supplied spellings; effective ancestor-global flags from outermost to innermost, each in declaration order; resolved-command flags in declaration order; then positionals in declaration order. Omitted optional fields contribute no tokens. This ordering does not claim that usage assigns preference to declaration order; it closes a boundary the artifact leaves open.

The artifact's four `double_dash` modes are preserved. `required` inserts one structural `--` immediately before the first affected positional. `optional` permits that structural token to be present or absent, selected at the `delimiter` point with neither spelling preferred. `automatic` inserts no token: after the first affected positional, subsequent flag-looking tokens are values under the artifact's own parse semantics. `preserve`, which usage admits only on a variadic positional, likewise inserts no structural token; every caller array element equal to the string `"--"` is emitted in place as an ordinary positional value, without insertion or removal. Configuration cannot contradict `required`, `automatic`, or `preserve`.

Credentials are not input fields. A processor MUST NOT place credential material on argv (**USAGE-P-06**). A usage descriptor declares no authentication scheme or credential-to-environment mapping, so this specification invents none: consumer configuration may supply explicitly named process environment variables, while a generic runtime credential without an environment-variable name is surfaced for consumer resolution rather than silently mapped. How a consumer acquires credentials or constructs the process environment is a runtime concern; the project's [`binding-invoker`](https://openbindings.com/interfaces/binding-invoker) interface defines one such negotiation surface (informative).

### 9.2. Configuration points

Six named configuration points cover choices the descriptor either leaves to invocation context or cannot declare (**USAGE-P-05**). What this specification pins at each is the artifact answer or fallback and what a consumer override _means_; the seam through which an override arrives is the implementation's own surface. Where an implementation exposes configuration, a per-invocation tier is consulted before a consumer-level tier and a declined override falls through. All fallbacks are content-independent: decided by declarations and framing, never by payload bytes.

| Point                        | Artifact answer or fallback                                                                                                                                                                                                                                      | Overrides                                                                                                                                                                                                             |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **target**                   | the descriptor's non-empty `bin` value                                                                                                                                                                                                                           | a complete executable address or path; it changes where that same artifact-declared binary identity is reached, never the command path or descriptor identity                                                         |
| **delimiter**                | the descriptor's `required`, `optional`, `automatic`, or `preserve` behavior, exactly as §9.1 defines                                                                                                                                                            | selects structural `--` present or absent only for `optional`; no OpenBindings preference is imposed                                                                                                                  |
| **encode** (per input field) | strings ride verbatim; no representation for other values                                                                                                                                                                                                        | supplies the deterministic one-token encoding of a non-string scalar occurrence; for stdin/file routing the same point may instead supply bytes. It never changes field identity, repetition, or argv structure       |
| **route** (per input field)  | argv, the artifact's highest-precedence carriage, per §9.1                                                                                                                                                                                                       | the artifact's declared environment variable; stdin (as a `-` operand where the surface declares one, or the pure stdin channel); or a temporary file whose path substitutes for the value; encoding remains separate |
| **decode**                   | the OpenBindings convention: standard output as text — bytes decode as UTF-8, invalid sequences a loud decode error — trailing line endings stripped (command-substitution semantics): repeatedly remove a final LF and an immediately preceding CR when present | any consumer-declared decoding of the completed process's output                                                                                                                                                      |
| **classify**                 | the OpenBindings convention: success **iff** exit status 0                                                                                                                                                                                                       | any consumer-declared classification of exit statuses                                                                                                                                                                 |

For a stdin or temporary-file route with no consumer-declared encoder, a string value contributes its UTF-8 bytes verbatim and every non-string value is refused. A configured `encode` choice may instead supply bytes for that one occurrence. A `-`-operand route writes those bytes to process stdin and substitutes the literal `-` for the field's ordinary argv value; a pure-stdin route writes the bytes without adding an operand. A temporary-file route writes the bytes to a file that remains readable for the process lifetime and substitutes its path for the field's ordinary argv value. Directory choice, permissions beyond being private to the invoking context, and cleanup mechanism are implementation security policy, not operation results.

An **environment route** is admissible only when the artifact declares `env` for that field; configuration names the route, never a replacement environment-variable name. It contributes no argv token and sets the declared variable for the spawned process. A string scalar is used verbatim; a Boolean no-argument flag uses the pinned usage parser's accepted `true`/`false` spellings; another scalar requires `encode` to produce exactly one string. Count, repeatable, and variadic surfaces cannot use the environment route in revision 1 because one environment string cannot preserve their OpenBindings occurrence structure under the pinned usage semantics. If two routed values would set one environment name differently, invocation refuses before spawn. An identically named value already supplied by consumer process-environment configuration must be equal or the invocation likewise refuses; silent precedence is not invented. A configured declared environment value may satisfy an omitted required field exactly as the artifact defines.

Routing replaces only the selected field's ordinary value contribution; it does not alter field identity, flag spelling, repetition, ordering, or artifact constraints. Revision 1 admits stdin and file substitution only for a single value-bearing occurrence. A repeatable or variadic field therefore cannot use those routes. An impossible routing — two fields routed to stdin, a `-`-operand route whose effective surface has no value slot, or a pure-stdin route that would leave a required positional or flag value unsatisfied — is refused before spawn. Per-CLI knowledge (a tool that emits JSON, alternate success exits, a field that must ride stdin) is packaged as consumer configuration at these points, never authored into the artifact or the OBI.

**Binary output** has no boundary encoding in revision 1: the decode default is text (UTF-8, invalid sequences a loud decode error), so a command emitting bytes that are not valid UTF-8 fails loudly rather than decoding to a guessed value. A consumer needing raw or non-text bytes overrides at the decode point; a base64 output lane is a declared gap this revision does not fill (the [catalog README](../README.md#the-bytes-boundary) records the pattern a later revision would follow). The same holds for a binary value routed to stdin or a temp file.

### 9.3. Success and the output value

Which outcomes of an invocation are successes is decided by the **classify** point. For a successful outcome, the operation's single output value is the product of the **decode** point applied to the process's standard output. Failure outcomes are not operation results and have no representation in this specification; what a consumer surfaces about them is its own concern.

### 9.4. Transform positions

This specification defines **no** context bindings at transform positions: a transform on a usage binding evaluates in the core's closed environment, unaugmented (OBI-B-02 item 7: the answer is _none_).

## 10. Conformance

Rules carry stable identifiers under the same discipline as the core's: never reused, never renumbered. Source rules bind OBI content governed by this specification; processor rules bind implementations claiming support for `openbindings.usage@1`. Verification follows the core's partial-verification posture.

- **USAGE-D-01**: `content`, when present, is a JSON string carrying UTF-8 usage descriptor source text.
- **USAGE-D-02**: `location`, when present, is a document address (absolute URI) or an exec address (`exec:` + space-separated argv vector), per [§4](#4-location).
- **USAGE-D-03**: `ref`, when present, is a non-empty space-separated command path resolving in the artifact's command tree per [§7](#7-ref); the root command is addressed by omitting `ref`.
- **USAGE-P-01**: Every descriptor is interpreted under usage v3.5.6, and `min_usage_version`, when present, is a SemVer load-time gate no greater than 3.5.6; a processor never selects its own usage line under this identifier, per [§3](#3-accepted-source-representations).
- **USAGE-P-02**: A processor MUST NOT dereference an exec address without explicit authorization; default deny, per [§4](#4-location).
- **USAGE-P-03**: The artifact's non-empty `bin` supplies target identity; `name` and the descriptor-acquisition address do not. Execution is one direct process spawn, never a shell, bound to the invocation lifetime, per [§8](#8-target-and-interaction).
- **USAGE-P-04**: Input fields use the canonical identities, type/arity rules, artifact constraints (including deterministic `choices env` resolution), four-mode delimiter behavior, and explicitly labeled OpenBindings argv-order convention of [§9.1](#91-input-mapping); strings ride as tokens verbatim, while ordinary non-string values require `encode`; collisions, unmatched fields, unsupported ambiguous multiplicity, and order-dependent overrides are refused before spawn.
- **USAGE-P-05**: The target, delimiter, encode, route, decode, and classify configuration points behave per [§9.2](#92-configuration-points): configuration cannot create missing artifact identity or contradict an artifact decision; non-string occurrences have no invented token/byte encoding; argv/environment/stdin/file routing and the explicitly labeled decode/classify conventions are as specified; impossible or colliding routings refuse before spawn. Where an implementation exposes configuration tiers, the more specific is consulted first and a declined override falls through. The carriage of a configuration value is implementation surface, not this rule's content.
- **USAGE-P-06**: Credential material never rides argv; explicitly named process environment variables come from consumer configuration, and the specification invents no mapping for generic credentials, per [§9.1](#91-input-mapping).
- **USAGE-P-07**: Descriptors containing upstream `include` or command `mount` composition are refused at load; revision 1 does not invent a portable filesystem base or execute dynamic artifact discovery, per [§3](#3-accepted-source-representations) and [§6](#6-composition).
- **USAGE-P-08**: Descriptors declaring the upstream configuration-file lane (`config`, `config_file`, `config_alias`, or field-level `config`) or an argument `parse` command are refused at load; revision 1 neither consults auxiliary files/processes nor ignores their declared precedence and parsing semantics, per [§3](#3-accepted-source-representations).

Conformance fixtures keyed to these identifiers are published with the project's conformance corpus. Deterministic _generation_ of OBI documents from usage descriptors (operation-key derivation from command paths, schema carriage) is a synthesis concern outside this specification; the project's interface-synthesizer and reference-tool documentation record those conventions.
