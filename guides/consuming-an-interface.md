# Consuming an OpenBindings Interface

**Status**: Implementation guidance. Not part of the normative specification.

This document describes the pattern for consuming an OpenBindings interface from a target service. It covers each constituent piece, its role, responsibilities, and dependencies. For the inverse ŌĆö how to *implement* a new binding format ŌĆö see [Implementing a Binding Format](./implementing-a-binding-format.md).

## Overview

The goal: a developer works with a typed object that knows what operations a service offers and how to reach them. The OBI tells the runtime how. Binding invokers handle the transport. The developer never writes protocol-specific code.

There are two layers:

- **Untyped runtime** ŌĆö the SDK's `OperationInvoker`. Works with any OBI. Operations are strings, input/output are untyped. Routes operations to binding invokers.
- **Typed wrapper** ŌĆö coupled to a specific interface. Operations are methods with known input/output shapes. Wraps the untyped runtime. Produced by codegen, written by hand, or generated by an AI agent ŌĆö the mechanism doesn't matter.

```typescript
// Untyped: works with any interface, untyped events
const operationInvoker = new OperationInvoker([new OpenAPIInvoker()]);
const iface = await fetchInterface("https://my-service.example.com");
for await (const event of operationInvoker.invoke({
  interface: iface,
  operation: "listItems",
  input: { limit: 10 },
})) { /* ... */ }

// Typed: coupled to a specific interface (codegen output)
const operationInvoker = new OperationInvoker([new OpenAPIInvoker()]);
const myService = new MyServiceInvoker(operationInvoker);
const iface = await fetchInterface("https://my-service.example.com");
const result = await myService.listItems(iface, { limit: 10 });
// result is typed ŌĆö the invoker knows the shape
```

Fetching the OBI does not require auth. If the service requires authentication for some or all operations, auth is a separate step (see [Auth Flow](#auth-flow)).

The typed invoker wraps the untyped runtime. Everything below (binding invokers, context store, OBI resolution) is the same for both.

## Pieces

### 1. OpenBindings Core SDK

**Role**: Foundation types and runtime primitives.

**Responsibilities**:
- Core types: `OBInterface`, `Operation`, `InvocationOutput`, `BindingContext`, `ContextStore`, `PlatformCallbacks`, etc.
- `OperationInvoker`: the runtime mux. Routes operation invocation to binding invokers by reading the OBI's bindings to find which invoker handles each operation. Applies transforms if configured.
- `fetchInterface(url)`: fetches an OBI from `<host>/.well-known/openbindings` (and follows redirects, validates the document).
- `ContextStore`: per-host context persistence. Keys are `host[:port]` (scheme-agnostic). Values are opaque maps with well-known field conventions for credentials (`bearerToken`, `apiKey`, `basic`) and transport (`headers`, `cookies`, `environment`).
- Schema normalization and compatibility checking (see conventions `OB-2020-12` subsumption profile).
- Validation (`validateInterface()`).

**Dependencies**: None. The SDK is the leaf dependency.

**Does NOT contain**: Binding invoker and creator implementations, codegen logic, CLI commands.

### 2. Binding Invokers and Interface Creators

**Role**: Binding format interpreters and transport handlers.

**Responsibilities**:
- Implement `BindingInvoker`: given a source (format + location), a ref, and input, make the protocol-specific call and return a stream of events. Unary calls produce one event; streaming calls produce many.
- Optionally implement `InterfaceCreator`: synthesize an OBI from a binding artifact (e.g., create an OBI from an OpenAPI spec).
- Optionally implement `SourceInspector`: enumerate refs available in a source (used by `ob` to populate binding tables).
- Read the binding specification document (OpenAPI spec, AsyncAPI spec, etc.) to understand endpoints, methods, parameters, security schemes.
- Apply credentials from the `BindingContext` according to the binding spec's security configuration. Each invoker determines the protocol-appropriate mechanism for credential transmission.
- Handle auth resolution: on auth failure, prompt for credentials via `PlatformCallbacks`, store them in `ContextStore` for reuse.

**Dependencies**: SDK (for types, context helpers, normalization utilities).

**Does NOT depend on**: Other binding invokers, typed invokers, or the CLI.

**Examples**:
- `@openbindings/openapi`: reads OpenAPI 3.x specs, makes HTTP requests, handles path/query/header/body parameter routing.
- `@openbindings/asyncapi`: reads AsyncAPI 3.x specs, handles SSE streams and WebSocket connections.
- `@openbindings/mcp`: connects to MCP servers via Streamable HTTP, invokes tools/resources/prompts.
- `grpc-go`: reads protobuf descriptors via server reflection, makes gRPC calls.
- `connect-go`: invokes Connect (Buf) RPCs over HTTP/1.1 with JSON payloads.

Each binding invoker advertises the format tokens it handles (e.g., `openapi@3.0`, `asyncapi@3.0`). The `OperationInvoker` matches operation bindings to invokers by format.

Format tokens are community-driven ŌĆö there is no central registry that gates them. Well-known formats use short names (`openapi`, `asyncapi`, `grpc`). Vendor- or project-specific formats use reverse-DNS naming to avoid collisions (e.g., `com.example.gateway-envelope@1.0`). Anyone can create a format token and a binding invoker that handles it.

### 3. Typed Operation Invoker

**Role**: Typed developer-facing API for a specific interface.

A typed operation invoker wraps the SDK's `OperationInvoker` and adds compile-time type safety for a specific interface's operations.

**Responsibilities**:
- Constructor takes a pre-built `OperationInvoker`. The developer assembles it with their chosen binding invokers, context store, and callbacks before passing it in.
- One typed method per operation: `getInfo(iface, input?)` returns `Promise<SoftwareInfo>`, etc. The interface is passed per call so a single typed invoker can dispatch against multiple providers of the same role contract.
- One stream method per operation (suffixed `Stream`): `getInfoStream(iface, input?)` returns `AsyncGenerator<TypedInvocationOutput<SoftwareInfo>>`.
- Embeds the interface contract as a `static readonly CONTRACT: OBInterface` constant. This is the "required interface" ŌĆö the typed invoker knows what operations it expects, and the constant can be checked against the live OBI for compatibility.
- Does NOT embed bindings, sources, or transforms. Those come from the live OBI at the target URL, passed per call.

**Dependencies**: SDK (for `OperationInvoker`, `OBInterface`).

**Does NOT depend on**: Any specific binding invoker. The developer configures the `OperationInvoker` with their chosen binding invokers before passing it to the typed invoker.

**How it's produced**: However the developer wants:
- `ob codegen <interface> --lang <language>` ŌĆö the CLI reads the interface, builds an IR, and emits language-specific code. Most common path.
- Hand-written ŌĆö a developer writes a class that wraps `OperationInvoker` with typed methods.
- AI-generated ŌĆö an agent produces the same code from the interface.

The mechanism is irrelevant. What matters is that the wrapper is typed to a specific interface.

### 4. OBI Document (at the service)

**Role**: The contract + binding instructions served by the target service.

**Responsibilities**:
- Declares operations with input/output schemas.
- Declares sources (binding artifacts: OpenAPI specs, AsyncAPI specs, etc.) with format and location.
- Declares bindings: maps each operation to a ref within a source.
- Optionally declares roles (interfaces this service satisfies) and transforms (schema bridges between operation schemas and binding schemas).
- Served at `/.well-known/openbindings` for discovery.

**The OBI is the bridge between the typed invoker and the binding invokers.** The developer calls `myService.listItems(iface, input)`. The typed method calls `operationInvoker.invoke({ interface: iface, operation: "listItems", input })`. The `OperationInvoker` looks up `listItems` in the OBI's bindings, finds it's bound to a source at a specific ref, finds a binding invoker that handles the source's format, hands it the source location and ref, and the binding invoker handles the protocol-specific work.

### 5. Context Store

**Role**: Per-host context persistence. The SDK-side surface of the `openbindings.context-store` role (`getContext` / `setContext` / `deleteContext`).

**Responsibilities**:
- Stores context keyed by `host[:port]` (no scheme ŌĆö `http://`, `https://`, `ws://` all share the same key for the same host).
- Values are opaque maps with well-known field conventions: credential fields (`bearerToken`, `apiKey`, `basic` with `{username, password}`, `accessToken`/`refreshToken`/`expiresAt`) and transport fields (`headers`, `cookies`, `environment`, `metadata`). Credentials are one kind of context; the store carries everything per-target an invoker needs.
- Binding invokers read from it before each request. The invoker merges stored context with per-call context; per-call values take precedence.
- Implementations: `MemoryStore` (in-memory, ephemeral), browser-persistence stores (LocalStorage / IndexedDB), OS-keychain-backed stores, hosted credential-vault clients.

**The developer stores a token once; every binding invoker for that host uses it automatically.**

### 6. Platform Callbacks

**Role**: Interactive credential resolution.

**Responsibilities**:
- `prompt(message, opts)`: ask the user for a credential (API key, password).
- `browserRedirect(url)`: open a URL for OAuth flows.
- `confirmation(message)`: yes/no dialog.
- `fileSelect(message, opts)`: let the user pick a file (client certificates, key files).
- Used by binding invokers when a request returns 401 and credentials need to be obtained interactively.

**The developer provides these to the `OperationInvoker` at construction time.** Browser apps implement them with modals/popups. CLI apps implement them with terminal prompts. Headless apps skip them (auth is pre-configured).

## Putting It Together

The typed invoker and the binding invokers both depend on the SDK but do not depend on each other. The developer composes them at runtime:

```typescript
import { OperationInvoker, MemoryStore, fetchInterface } from "@openbindings/sdk";
import { OpenAPIInvoker } from "@openbindings/openapi";
import { MyServiceInvoker } from "./generated/my-service-invoker.js";

const operationInvoker = new OperationInvoker(
  [new OpenAPIInvoker()],
  { contextStore: new MemoryStore() },
);
const myService = new MyServiceInvoker(operationInvoker);

const iface = await fetchInterface("https://api.example.com");
await operationInvoker.contextStore?.set("api.example.com", { bearerToken: "..." });
const menu = await myService.getMenu(iface);
```

The typed invoker knows the interface. The binding invokers know the protocols. Neither knows about the other. The SDK provides the primitives both build on.

## Unified Invocation Model: Everything Is a Stream

An OpenBindings interface defines operations ŌĆö not request/response pairs, not subscriptions, not RPC calls. Just operations with input and output schemas. The interface doesn't say whether an operation yields one result or many. It can't, because that depends on the binding: the same operation might return a single JSON response over REST and a continuous event stream over AsyncAPI.

This means the invocation model must handle both cases uniformly. Every operation invocation returns a **stream of events**. Each event carries either output or an error:

```typescript
for await (const event of myService.getMenuStream(iface)) {
  if (event.error) { /* handle error */ }
  if (event.output) { /* handle output */ }
}
```

For unary operations, the stream yields one event and closes. For subscriptions, it yields events until the connection closes or the consumer cancels. The consumer code is the same ŌĆö iterate until done.

This is not a design choice for convenience. It's a consequence of protocol agnosticism. If the invocation model assumed request/response, it couldn't represent streaming. If it assumed streaming, unary would be a degenerate case (which it is ŌĆö a stream of one). The stream model is the only one that works for all bindings without the interface having to declare which pattern applies.

**Typed invokers provide a unary convenience** ŌĆö `await myService.getMenu(iface)` returns `Promise<MenuResponse>` by taking the first event from the stream and throwing on error. This is sugar for the common case where the developer knows the operation is unary. The stream variant (`myService.getMenuStream(iface)`) is always available for operations that may yield multiple events or when the developer wants event-level control.

## Auth Flow

`fetchInterface(url)` resolves the OBI. It does not require authentication. After fetching, the developer can call any operation ŌĆö unauthenticated operations work immediately.

If the service requires auth for some or all operations, auth can be handled proactively or resolved interactively:

1. Developer calls `fetchInterface(url)` ŌĆö resolves the OBI, no token needed.
2. If the OBI declares `security` entries on its bindings, the SDK knows what authentication methods are available.
3. When an operation fails with `auth_required`, the binding invoker walks the security methods in preference order and uses platform callbacks (`prompt`, `browserRedirect`) to acquire credentials interactively.
4. Acquired credentials are stored in the `ContextStore` keyed by the source endpoint.
5. The binding invoker retries the operation once with the new credentials.
6. Subsequent operations automatically include the stored credentials.

If the developer already has a token (e.g., from a previous session), they can put it in the context store directly before invoking:

```typescript
await operationInvoker.contextStore?.set("api.example.com", { bearerToken: existingToken });
```

**Auth is optional.** If the service doesn't require auth, skip steps 2ŌĆō4. If the service allows unauthenticated access to some operations, those work immediately. Operations that require auth either fail with a clear error or trigger interactive resolution via `PlatformCallbacks`.

The developer stores a token once. Every binding invoker, every protocol, every request for that host uses it automatically.

## Roles and Conformance

An interface can declare that it satisfies roles (published shared interfaces):

```json
{
  "roles": {
    "openbindings.context-store": "https://openbindings.com/interfaces/context-store.json"
  },
  "operations": {
    "getContext": {
      "satisfies": [{ "role": "openbindings.context-store", "operation": "getContext" }]
    }
  }
}
```

Clients can check conformance at runtime: "does this service play the context-store role?" This enables the delegate pattern across service boundaries ŌĆö target the role, not the service.

Typed invokers are per-interface. If you produce an invoker from the context-store interface, you get a `ContextStoreInvoker` with `getContext(iface, ŌĆ”)`, `setContext(iface, ŌĆ”)`, etc. Any service that conforms to that role works with the same typed invoker ŌĆö you simply hand it the live OBI of the service you want to call.

## Key Principles

1. **The developer never writes protocol-specific code.** Typed invoker + binding invokers + OBI handle transport.
2. **Binding invokers are pluggable.** The developer chooses which formats to support. No unnecessary dependencies.
3. **The OBI is the bridge.** It connects the typed invoker's operations to the binding invoker's transport capabilities.
4. **Credentials are per-host, not per-request.** Store once, use everywhere.
5. **Everything is a stream.** Unary operations are streams of one event. The pattern is the same for request/response and real-time subscriptions.
6. **Composition at runtime.** Typed invoker + binding invokers are composed when the typed invoker is constructed, not when the code is produced.
7. **Interface is passed per call.** A single typed invoker can dispatch against multiple providers of the same role contract ŌĆö the developer hands it the live OBI of whichever service they want to call.
